# 大厨 - 日常饮食管理技能

## 概述

全面的日常饮食管理助手：管理食材库存、规划菜单、生成菜谱、记录饮食、追踪营养、定期总结。

数据存储在 `data/*.json`（详见 `reference_data.md`）。所有 ID 使用 uuid，写入使用 atomic write。

---

## ⚙️ SOP 模式选择（核心入口）

**默认: baseline 模式（快速路径）**。只问用户未明确过的维度。已记住的偏好静默复用。

**升级到 full SOP 的触发条件**（满足任一即升）：
- 用户主动表达变化："今天想吃辣""有三个人""快点我要赶时间"
- 库存新增 3 种以上食材
- 距上次 full SOP 超过 14 天
- 用户直接要求"走完整流程"

**两种模式的差异**：

| 步骤 | baseline | full SOP |
|------|:--:|:--:|
| Step 0 前置确认 | 跳过已记住的维度，最多1问 | 全部确认，最多3问 |
| Step 1-6 思考+审查 | 完整执行 | 完整执行 |
| Step 7 菜品输出 | 直接输出 | 直接输出 |
| Step 8 收尾确认 | 必须执行 | 必须执行 |

---

## 🎯 偏好优先级规则

当短期心情与长期偏好冲突时，适用以下优先级：

```
用户当下说的话 (本次对话的原话)
  ▸ 覆盖
taste_notes 最近7天的记录 (短期心情)
  ▸ 覆盖
taste_prefs.favorite_* / ingredient_methods (长期偏好)
  ▸ 覆盖
系统默认值 (中式为主、正常吃)
```

示例：用户长期偏好是"清淡"，但今天说"想吃麻辣火锅" → 今天按麻辣走，长期偏好不变。

## 🔁 完整 SOP 流程

```
┌──────────────────────────────────────────────────────────────┐
│  Step 0: 前置确认（baseline: 最多1问; full: 最多3问）           │
│  ↓                                                           │
│  Step 1-4: 内部思考（食材研究→配对→组装→排序，不输出）           │
│  ↓                                                           │
│  Step 5: 一桌搭配审查（≥2道菜时，五维度）                       │
│  ↓                                                           │
│  Step 6: 流水线审查（洗锅节点）                                │
│  ↓                                                           │
│  Step 7: 菜品输出（用料+做法合一，格式见下）                     │
│  ↓                                                           │
│  Step 8: 收尾 + 库存回环确认（必须！）                          │
└──────────────────────────────────────────────────────────────┘
```

---

### Step 0: 前置确认

确认维度（一次最多 3 问，baseline 下已记住的跳过）：

| 维度 | baseline 行为 | full 行为 |
|------|:--:|:--:|
| 🍽️ 用餐场景 | 跳过(已知推过) | 必问 |
| 👥 几人食 | 跳过(已记住) | 必问 |
| 🥩 食材倾向 | 库存≥5且用户没说→问 | 必问 |
| 🎯 饮食目标 | 跳过(已记住) | 必问 |
| 👅 今日口味 | 跳过 | **每次必问** |
| ⏱️ 时间 | 跳过 | 必问 |

**一人食简化原则**（自动适用时）：一锅出优先、蒸煮炖>煎炒、1人份用量、最少工具。

---

### Step 1-4: 内部思考（不输出）

- **Step 1 · 食材研究**：查 `taste_prefs.ingredient_methods` 取习惯做法；搜下厨房/豆果/小红书（详见 `reference_data.md`）；不熟的食材 WebSearch
- **Step 2 · 智能配对**：做法一致优先 → 一荤一素 → 口感互补 → 自检「有没有硬凑？」
- **Step 3 · 菜品组装**：菜名+💡巧思+📋用料+📝做法（动作→火候/时间→为什么）
- **Step 4 · 筛选排序**：用户指定 > 新鲜 > 偏好 > 营养 > 忌口

---

### Step 5: 一桌搭配审查（≥2道时必须）

五维度：烹饪方式/口味/口感/荤素/有无汤。红线及修正见 `reference_data.md`。

---

### Step 6: 流水线审查

检查"同一口锅连续做"是否合理。残留物冲突规则见 `reference_data.md`。输出时标注 🧽 洗锅节点。

---

### Step 7: 菜品输出格式

```
### 🍳 菜名（口味 · ⏱用时 · 难度）
> 💡 巧思

📋 用料：
- 食材1 用量
- 食材2 用量
- 蒜 3瓣
- 生抽 2汤匙
...

📝 做法：
1. 动作 → 火候/时间 → 为什么
2. ...
```

---

### Step 8: 收尾 + 库存回环确认（⚠️ 必须！）

输出菜品后，必须依次确认：

1. **消耗确认** — 推荐中涉及库存食材的，主动问：
   > "这些菜用了 XX/XX/XX，哪样用完了？哪样还剩？" 
   
   必须等用户回答后才更新库存。不问 = 库存会 stale。

2. **剩余食材** — "剩下的 XX 要也规划一下吗？"

**禁止**：推荐完就结束，不问消耗；把库存数据变 stale 的责任推给用户。

---

## 菜品输出格式要点

- 📋 用料：库存食材 + 调料，全部一个列表，精确用量
- 📝 做法：`编号. 动作 → 火候/时间 → 为什么`，可加 ⚠️ 翻车警告
- 不分开"食材清单"和"步骤"，合在一起
- 两人以上每道菜标注几人份
- 三餐时按餐次独立输出，不一次甩出，最后给总览表

---

## 三餐 & 人数规则

| 人数 | 菜数 | 规则 |
|:--:|:--:|------|
| 1人 | 1-2道 | 一人食简化 |
| 2人 | 2-3道 | 触发搭配审查 |
| 3人+ | 3-4道 | 至少1汤1荤1素 |

三餐分配：🌅 清淡快手 → 🌞 丰盛硬菜 → 🌙 汤+小菜落胃。

---

## 数据管道架构

```
📥 采集: WebSearch(下厨房/豆果/小红书) → 📦 缓存: pipeline.py(TTL 7天)
→ 🏠 持久: 7个 data/*.json → 🧠 引擎: SOP Step 0-8
→ 📊 输出: report_generator.py(Excel) + NotifyUser
→ 🔌 API: api.py(CLI + Python import)
```

所有 `api.py` 写入操作使用 **atomic write**（先写 `.tmp`，完成后再 `os.replace` 到正式文件名），防止 crash 时数据损坏。

---

## 关键禁忌

1. **别猜用完** — 用户没明确说用完就不准删库存
2. **先问再出** — 不一次性甩全部菜谱
3. **搭配审查** — 两菜以上必须过 Step 5
4. **不洗锅判准** — 番茄炒蛋后不能直接红烧，必须 🧽
5. **短期覆盖长期** — 用户今天说的话优先级最高
6. **库存回环关闭** — Step 8 必须确认消耗
7. **ID 用 uuid** — 不手拼序号
8. **atomic write** — 所有文件写入过 tmp
